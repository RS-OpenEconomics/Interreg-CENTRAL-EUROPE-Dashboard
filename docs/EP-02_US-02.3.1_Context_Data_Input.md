## US-02.3.1 — Context Data Numeric Input

**Epic:** EP-02 — Need Description and Baseline Scenario
**Feature:** F-02.3 — Quantitative Context Data
**Priority:** Must Have
**Dependencies:** US-01.3.1 (a need must be selected)

### 1. What this screen/component does
Step 2.3 page ("Dato di contesto" in the sidebar). Displays a single numeric input field whose label is dynamic, determined by the `q1_label` field of the need selected in Phase 1. The field is optional — the officer can advance without filling it.

### 2. Purpose
Allow the officer to provide a quantitative indicator that contextualises the magnitude of the need. The value feeds the Scenario Zero section of the DOCFAP report as supporting evidence. It does not participate in CBA, MCA, or any calculation.

### 3. Input data

| Field | Type | Required | Source |
|---|---|---|---|
| Context data value | Number (> 0) | No | User input |

### 4. Pre-populated or default data
- **Label:** read from the `q1_label` field of the selected need (OpenCore Layer 1 → Needs). Example: "Stato di conservazione del bene (classe conservazione MiC)" for FAB-31. If the need has no `q1_label`, the default label is "Dato quantitativo di contesto".
- **Value:** empty on first access. If the user returns, the previous value is restored from the store.

### 5. Business rules and validations
- The field is optional — the step can advance with the field empty.
- If a value is entered, it must be a number greater than zero. If the value fails validation, an inline error is shown: "Se inserito, il valore deve essere un numero maggiore di zero."
- An invalid (non-empty, non-valid) value blocks advancement. Clearing the field removes the error and allows advancement.
- A helper text is always visible: "Dato facoltativo — inserisci il valore numerico dell'indicatore. Puoi procedere anche senza compilare questo campo."

### 6. UI element behaviour

| Element | Type | Initial state | Behaviour | Effect |
|---|---|---|---|---|
| Dynamic label | Label element | Shows `q1_label` from selected need (or default) | Read-only | — |
| Helper text | Static text | Always visible | Read-only | — |
| Numeric input | Number input (min=1, step=1) | Empty | User types | Validates; updates store `q1Value` |
| Validation error | Inline error | Hidden | Shown when value is entered but invalid (≤ 0 or NaN) | — |
| "Next" button | Navigation button | Active (field is optional) | Click | Advances to Phase 3 |

### 7. Frontend → OpenCore flow

```
Step 2.3 loads
  → Read q1_label from selected need (already available in store or cached from Phase 1)
  → Render input with dynamic label

User enters or clears value
  → Local state update: q1Value (number | null)
  → No API call
```

### 8. OpenCore entities involved

| Entity | Table | Usage |
|---|---|---|
| Need | Layer 1 → Needs | `q1_label` field read for the dynamic input label |

### 9. Edge cases and conditional behaviour
- **Need has no `q1_label`:** the label falls back to "Dato quantitativo di contesto". The field still appears and functions identically.
- **User enters zero or negative:** inline error shown. Clearing the field removes the error.
- **User enters a decimal:** accepted (the input type is `number` with no integer constraint beyond > 0).
- **Need change after filling:** if the user navigates back to Phase 1 and changes the need, `q1Value` is cleared from the store and the label updates to the new need's `q1_label`.

### 10. Acceptance Criteria

```gherkin
Scenario: Dynamic label from need
  Given the selected need has q1_label = "Indice di non autosufficienza e lista attesa RSA"
  When Step 2.3 loads
  Then the input label shows "Indice di non autosufficienza e lista attesa RSA"

Scenario: Default label when q1_label is absent
  Given the selected need has no q1_label
  When Step 2.3 loads
  Then the input label shows "Dato quantitativo di contesto"

Scenario: Optional field — advance without value
  Given the officer is on Step 2.3
  When the input field is empty
  Then the "Next" button is active
  And the officer can advance

Scenario: Valid numeric entry
  Given the officer is on Step 2.3
  When they enter a positive number
  Then q1Value is updated in the store
  And no error is shown

Scenario: Invalid entry blocks advance
  Given the officer is on Step 2.3
  When they enter 0 or a negative number
  Then an inline error "Se inserito, il valore deve essere un numero maggiore di zero" is displayed
  And the step cannot advance

Scenario: Clearing invalid value allows advance
  Given the officer entered an invalid value
  When they clear the field
  Then the error disappears
  And the step can advance with q1Value = null
```

### 11. Open notes
- [ ] Confirm whether the `step` attribute on the number input should be `1` (integers only) or `any` (allow decimals). The current POC uses `step=1` but some q1_label indicators may require decimal precision (e.g., conservation class indices).
- [ ] The future evolution of this step (pre-calculated indicator values with cluster gap comparison) is documented separately and is out of scope for this US.
